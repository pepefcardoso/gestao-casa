## UC01 — Configure Property and Rooms

**Actor:** User
**Platform:** Mobile (Expo) — first-time setup
**Related module:** Module 1

### Main Flow

1. User opens the app for the first time and sees a welcome screen prompting house registration.
2. User fills in house name (required) and optional fields: location, total area.
3. System creates a `houses` record and navigates to the rooms sub-screen.
4. User taps "+" and adds rooms one by one (name required, area and color optional).
5. System saves each `rooms` record linked to the house via `house_id`.

### Business Rules

- `area` (house and room) must be `> 0` if provided. Zod must reject zero and negative values with a field-level error message.
- `house.id` and `room.id` are UUID v4, generated by the database (`defaultRandom()`).
- `color_code` is a CSS hex string (e.g. `#3b82f6`). No validation beyond string format — UI provides a picker.
- A house may have zero rooms initially; rooms can be added at any time.
- Deleting a house cascades to all its rooms and their linked financing record.
- Deleting a room sets `room_id` to `null` on all linked expenses (`set null`).

### Error States

| Condition        | Response                                                 |
| ---------------- | -------------------------------------------------------- |
| Room area ≤ 0    | `border-orange-500` on field + inline validation message |
| House name empty | Block save, highlight field                              |

## UC02 — Generate and Adjust Financing

**Actor:** User
**Platform:** Web (Next.js)
**Related module:** Module 3
**Shared logic target:** `libs/shared-logic/src/utils/calculate-financing.ts`

### Main Flow

1. User accesses the Financing tab (`/financing`).
2. Inputs: property value, down payment, term (months), annual interest rate, amortization system (`SAC` or `PRICE`).
3. System derives `financed_amount = property_value - down_payment` and calls `calculateFinancing()`.
4. System renders a 360-row table (sticky header, `font-mono tabular-nums`, columns: month, installment, interest, amortization, outstanding balance).
5. An area chart above the table renders two layers (interest decay + amortization growth) across all 360 months.
6. User optionally fills "Real Value of 1st Installment" and/or "Real Value of Last Installment" sticky inputs beside the boundary rows.
7. System re-calls `calculateFinancing()` with overrides and updates the table and chart reactively.
8. User saves. System persists to `financing` table (one record per house — `house_id` is unique).

### Business Rules

- `down_payment` must be `≥ 0` and `< property_value`.
- `term_months` must be between 1 and 360.
- `interest_rate` is stored and passed as a decimal (e.g. `0.10` for 10% p.a.). Monthly rate derived inside `calculateFinancing()`.
- Override logic (see `mvp-spec.md` §Module 3): overrides lock their row; delta is distributed across intermediate rows 2…N-1.
- A house can have at most one `financing` record. Saving a second time updates (`upsert`) the existing record.
- `calculateFinancing()` must be a pure function — no DB calls, no side effects. Input/output types defined in schema.

### Error States

| Condition                       | Response                             |
| ------------------------------- | ------------------------------------ |
| `down_payment ≥ property_value` | Inline error on `down_payment` field |
| `term_months` out of range      | Inline error on field                |
| API save fails                  | Toast error, form state preserved    |

## UC03 — Register Expense or Budget

**Actor:** User
**Platform:** Mobile (Expo) — primary; Web supports read + status toggle
**Related module:** Module 2

### Main Flow

1. User taps the floating "+" button.
2. Fills in: description (required), total amount (required), category (required), priority (required).
3. Toggles status: `Budget` (still evaluating) or `Confirmed` (purchased/committed).
4. Selects payment type:
   - **Upfront:** `installments_count = 1`, single `due_date`.
   - **Installments:** slider (1–24) + text input (up to 360); live preview shows `R$ {total} ÷ {n} = R$ {per_month}/mês`.
5. Optionally links to a room (`room_id`).
6. Saves. System creates `installments_count` expense entries, each with `total_amount / installments_count`, starting from `due_date` and incrementing by 1 month.

### Business Rules

- `total_amount` must be `> 0`.
- `installments_count` must be between 1 and 360.
- `due_date` is the date of the **first** installment. Subsequent entries are `due_date + N months`.
- Installment description suffix: `"{description} (1/N)"`, `"{description} (2/N)"`, etc.
- `BUDGET` status: entries appear in Cash Forecast chart only.
- `CONFIRMED` status: entries appear in both Actual Expenses and Cash Forecast.
- Status toggle `BUDGET → CONFIRMED` updates **all** sibling installment entries atomically.
- `room_id` is nullable. If the linked room is deleted, `room_id` is set to `null` (no cascade delete on expense).

### Error States

| Condition                         | Response                                   |
| --------------------------------- | ------------------------------------------ |
| `total_amount ≤ 0`                | Block save, inline validation              |
| `installments_count` out of range | Clamp slider at boundary, show helper text |
| No description                    | Block save, highlight field                |

## UC04 — View Financial Dashboard

**Actor:** User
**Platform:** Web (Next.js)
**Route:** `/dashboard`

### Main Flow

1. User opens the dashboard.
2. System displays a 12-month horizontal grid. Each month column shows:
   - Financing installment for that month (from `financing` record).
   - Sum of `CONFIRMED` expense installments due that month.
   - Sum of `BUDGET` expense installments due that month (visually distinct).
   - **Total outflow:** `financing_installment + confirmed_installments + budget_installments`.
3. User can click a month column to drill into the expense list filtered by that month.

### Business Rules

- Financing installment per month is read from the pre-calculated schedule (do not re-run `calculateFinancing()` on render — use persisted data or memoized output).
- `BUDGET` entries are displayed with `bg-amber-600` semantic token; `CONFIRMED` with `bg-rose-600`.
- If no financing record exists for the house, the financing row shows `—`.
- Dashboard shows the **next** 12 months from the current date, not a fixed calendar year.

## UC05 — Toggle Expense Status (Budget → Confirmed)

**Actor:** User
**Platform:** Web (Next.js) — month-end review flow; Mobile — quick action
**Related module:** Module 2

### Main Flow

1. User locates a `BUDGET` expense entry (on dashboard or expense list).
2. Taps/clicks the status toggle.
3. System updates all sibling installment entries (same description group) from `BUDGET` to `CONFIRMED` atomically.
4. Charts and totals update immediately via optimistic UI.

### Business Rules

- Toggle is **irreversible** in the current MVP scope (no `CONFIRMED → BUDGET` downgrade).
- All entries in the installment group must be updated in a single DB transaction.
- Optimistic update: apply status change in local state immediately; roll back on API error.
- Color transition on the card: `bg-amber-600 → bg-rose-600` (CSS transition, not a hard re-render).

### Error States

| Condition        | Response                                     |
| ---------------- | -------------------------------------------- |
| API update fails | Roll back optimistic state, show toast error |

## Field Validation Summary

> For Zod schema implementation, derive all schemas from `libs/backend/src/db/schema.ts` using `createInsertSchema`. Apply refinements as `.refine()` calls on the insert schema — never inline inside route handlers.

| Field                        | Rule                        | Zod method                        |
| ---------------------------- | --------------------------- | --------------------------------- |
| `house.name`                 | non-empty string            | `.min(1)`                         |
| `room.area`                  | positive number if provided | `.positive()` on optional numeric |
| `expense.total_amount`       | > 0                         | `.positive()`                     |
| `expense.installments_count` | 1–360                       | `.int().min(1).max(360)`          |
| `financing.down_payment`     | ≥ 0 and < `property_value`  | `.refine()` cross-field check     |
| `financing.term_months`      | 1–360                       | `.int().min(1).max(360)`          |
| `financing.interest_rate`    | > 0                         | `.positive()`                     |
